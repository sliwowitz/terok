Credential proxy design to prevent secret exfiltration from task containers.

**Why:** Shield (nftables firewall) blocks domains but can't prevent exfiltration to allowed domains like github.com. Current shared mounts expose real API keys and OAuth tokens.

**How to apply:** Two design docs in `docs/brainstorming/`:
- `credential-proxy-socket.md` — primary approach using vendor base URL env vars + Unix socket reverse proxy
- `credential-proxy-mitm.md` — alternative using mitmproxy TLS interception (custom mitmproxy or Sandcat/WireGuard)

**Key decisions (2026-03-24):**
- Socket approach first, present to Podman community
- Auth containers use temp mounts (not shared mounts) → shared mounts never contain secrets
- Phantom API keys make tools use "API key mode," skipping OAuth entirely in-container
- Credential store: JSON files per-project, upgrade to keyring later
- SSH agent forwarding (isolated per-project) replaces private key mounting
- One proxy daemon, one Unix socket, per-task phantom tokens
- Per-project auth supported from day one (credential sets)
- MITM is the fallback if socket approach proves infeasible for specific tools

**Verified env vars (from binary analysis):**
- Claude: `ANTHROPIC_BASE_URL`, `ANTHROPIC_UNIX_SOCKET`, `CLAUDE_CODE_CUSTOM_OAUTH_URL`
- Codex: `OPENAI_BASE_URL`, `CODEX_REFRESH_TOKEN_URL_OVERRIDE`
- gh: `http_unix_socket` config key
- glab: `api_host` + `api_protocol` config keys
# Credential Proxy: Socket Approach — Implementation Plan

## Context

terok currently bind-mounts vendor config directories (containing API keys,
OAuth tokens, SSH private keys) into task containers.  A prompt-injected or
supply-chain-compromised agent can read and exfiltrate these secrets to any
allowed domain.  This plan implements a Unix-socket-based credential proxy
that ensures no real secret ever enters a task container.

Full design: `memory/credential-proxy-socket-design.md`

## Package Boundaries

The credential proxy spans two packages:

**terok-sandbox** (container plumbing — knows nothing about specific agents):
- Credential proxy server (HTTP-over-Unix-socket reverse proxy)
- Proxy lifecycle management (systemd units, daemon start/stop)
- Credential store (generic JSON files, provider-agnostic)
- Proxy token CRUD (phantom token registry)
- SSH agent isolation (per-project ssh-agent management)

**terok-agent** (agent-specific knowledge — YAML registry + scripts):
- Auth interceptor (temp container mounts, credential extraction per tool)
- Per-agent credential metadata in YAML registry (credential file locations,
  extractor logic, base URL env vars, proxy route config)
- Environment variable injection (phantom tokens, base URLs per agent)

The split follows the existing pattern: terok-sandbox provides `RunSpec`,
gate server, shield, SSH — generic container services.  terok-agent provides
agent definitions, auth flows, wrapper scripts — agent-specific orchestration.

The YAML agent registry (e.g. `claude.yaml`) would be extended with a
`credential_proxy:` section:
```yaml
credential_proxy:
  route_prefix: claude
  upstream: api.anthropic.com
  auth_header: Authorization
  auth_prefix: "Bearer "
  phantom_env:
    ANTHROPIC_API_KEY: true
  base_url_env: ANTHROPIC_BASE_URL
  credential_file: .credentials.json   # relative to auth mount
  credential_type: oauth                # oauth | api_key
  oauth_token_endpoint: "https://console.anthropic.com/v1/oauth/token"
```

terok-sandbox runs a generic proxy reading route config from a JSON file
(generated by terok-agent from the YAML registry).  terok-agent handles
all agent-specific knowledge.

## Phased Implementation

The work is split into independently testable phases.  Each phase produces
working, committable code.  Later phases build on earlier ones.

**Security invariant:**  Once the credential proxy is implemented and enabled,
there is NO silent fallback to direct credential mounting.  If the proxy is
unreachable or misconfigured, the task **fails to start** with an actionable
error — same as the gate server's `ensure_server_reachable()` pattern.
During development, the proxy can be disabled via explicit config
(`credential_proxy.enabled: false`), but this is an opt-in user decision,
never an automatic degradation.

**Path mapping** (plan was originally written with terok paths — remap):
- `src/terok/lib/security/credential_store.py` → `src/terok_sandbox/credential_store.py`
- `src/terok/lib/security/proxy_tokens.py` → `src/terok_sandbox/proxy_tokens.py`
- `src/terok/credential_proxy/` → `src/terok_sandbox/credential_proxy/`
- `src/terok/lib/security/credential_proxy.py` → `src/terok_sandbox/credential_proxy_lifecycle.py`
- `src/terok/lib/security/ssh_agent.py` → `src/terok_sandbox/ssh_agent.py`
- `src/terok/lib/security/auth.py` → `src/terok_agent/auth.py`
- `src/terok/lib/containers/environment.py` → `src/terok_agent/` (env injection)
- `src/terok/lib/containers/headless_providers.py` → `src/terok_agent/headless_providers.py`
- `src/terok/cli/commands/credential_proxy.py` → `src/terok_sandbox/commands.py` (extend)
- YAML registry extensions → `src/terok_agent/resources/agents/*.yaml`

### Phase 1: Credential Store + Proxy Tokens

**Goal:** Host-side storage for captured credentials and per-task phantom tokens.

**New files:**

1. `src/terok_sandbox/credential_db.py` (~60 lines)  *[terok-sandbox]*
   - Single sqlite3 database at `state_root() / "proxy" / "credentials.db"`
   - WAL mode for concurrent multi-process access (no manual fcntl locking)
   - Two tables:
     - `credentials (credential_set TEXT, provider TEXT, data TEXT, PRIMARY KEY (credential_set, provider))`
       — `data` is JSON string with type-specific fields
     - `proxy_tokens (token TEXT PRIMARY KEY, project TEXT, task TEXT, credential_set TEXT)`
   - `CredentialDB` class (~30 lines for CRUD):
     - `store_credential(credential_set, provider, data_dict)`
     - `load_credential(credential_set, provider) -> dict | None`
     - `list_credentials(credential_set) -> list[str]`
     - `delete_credential(credential_set, provider)`
     - `create_proxy_token(project, task, credential_set) -> str`
     - `lookup_proxy_token(token) -> dict | None`
     - `revoke_proxy_tokens(project, task)`
   - Encryption upgrade path: wrap `data` with `cryptography.fernet` before
     INSERT, or swap `sqlite3` for `sqlcipher3` (drop-in replacement)

2. `tests/unit/test_credential_db.py`

**Why sqlite3 over JSON files:**
- Zero external deps (stdlib)
- Built-in WAL concurrency (no fcntl/tempfile/os.replace boilerplate)
- Atomic writes out of the box
- list() is trivial (SELECT)
- ~60 lines total vs ~230 for JSON+fcntl credential_store + proxy_tokens

```python
# BEFORE (line 262):
host_dir = envs_base / provider.host_dir_name

# AFTER:

    # ... run container with tmp mount ...
```

### Phase 3: Proxy Server Core

**Goal:** A standalone HTTP-over-Unix-socket reverse proxy that injects
credentials based on phantom tokens and path-prefix routing.

**New files:**

1. `src/terok/credential_proxy/__init__.py`
2. `src/terok/credential_proxy/__main__.py` — entry point (like `gate/__main__.py`)
3. `src/terok_sandbox/credential_proxy/server.py` (~150 lines)
   - **Zero terok imports** (same principle as `gate/server.py`)
   - Uses aiohttp: `web.UnixSite` for Unix socket listener,
     `aiohttp.ClientSession` for upstream forwarding + OAuth refresh
   - Shorter + more robust than stdlib for a streaming reverse proxy
     (native async SSE, built-in routing, graceful disconnect handling)
   - Startup: receives `--socket-path`, `--db-path`, `--routes-file`
   - Route config (JSON): maps path prefix → upstream base URL + auth header name
     ```json
     {
       "claude": {"upstream": "https://api.anthropic.com", "auth_header": "Authorization", "auth_prefix": "Bearer "},
       "codex":  {"upstream": "https://api.openai.com",    "auth_header": "Authorization", "auth_prefix": "Bearer "},
       "gh":     {"upstream": "https://api.github.com",    "auth_header": "Authorization", "auth_prefix": "token "},
       ...
     }
     ```
   - Request flow:
     1. Parse path: `/{provider}/{rest}` → extract provider prefix + remaining path
     2. Extract phantom token from `Authorization` or `X-Api-Key` header
     3. Lookup token in token file (lazy-load with mtime check, same as gate server)
     4. Load real credential from credential dir (JSON files)
     5. If OAuth + expired: refresh (POST to token endpoint, save new tokens)
     6. Strip phantom auth headers, inject real auth header
     7. Forward request to upstream, stream response back via `iter_any()`
   - Signal handling: SIGTERM → graceful shutdown

4. `src/terok/credential_proxy/resources/` — systemd unit templates
   - `terok-credential-proxy.socket`
   - `terok-credential-proxy.service`

5. `tests/unit/credential_proxy/test_server.py`
   - Test routing, phantom token validation, header injection
   - Test SSE streaming (mock upstream)
   - Test OAuth refresh logic

**Important:** The server has zero terok imports.  It reads credentials from
a sqlite3 DB and routes from a JSON file, both passed via CLI args — same
self-contained principle as `gate/server.py`.

### Phase 7: OAuth Token Refresh

**Goal:** The proxy can refresh expired OAuth tokens without container involvement.

**Modified files:**

1. `src/terok/credential_proxy/server.py`
   - Add `_refresh_oauth_token(cred_entry, cred_path)` function:
     ```python
     async def _refresh_oauth_token(entry, cred_path):
         async with aiohttp.ClientSession() as s:
             async with s.post(entry["token_endpoint"], data={
                 "grant_type": "refresh_token",
                 "refresh_token": entry["refresh_token"],
                 "client_id": entry.get("client_id", ""),
             }) as resp:
                 data = await resp.json()
                 entry["access_token"] = data["access_token"]
                 entry["refresh_token"] = data.get("refresh_token", entry["refresh_token"])
                 entry["expires_at"] = time.time() + data.get("expires_in", 28800)
                 # Atomic write back to credential file
                 _atomic_write_json(cred_path, entry)
                 return entry["access_token"]
     ```
   - Call before forwarding if `expires_at < time.time() + 300`
   - Token endpoints:
     - Claude: `https://console.anthropic.com/v1/oauth/token`
     - Codex: standard OpenID Connect (exact URL from auth.json or well-known)
     - gh: `https://github.com/login/oauth/access_token` (or PAT = no refresh)

## Dependencies

- **aiohttp** — one dep covering both Unix socket server and upstream HTTP
  client.  Native async streaming, built-in routing, connection pooling.
  Produces ~100 fewer LoC than stdlib `http.server` for this use case.
- The credential store uses stdlib `sqlite3` (WAL mode) — zero deps.

## Prior Art

Two projects validate this architecture:

- **nono** (github.com/always-further/nono) — Rust, Apache-2.0, 1.2k stars.
  Kernel-enforced AI agent sandbox with built-in credential injection reverse
  proxy.  Almost identical architecture: path-prefix routing, phantom session
  tokens, keyring-backed credentials, SSE streaming.  Built-in profiles for
  Claude Code, Codex, OpenCode.  Not reusable as dependency (coupled to nono's
  Landlock/Seatbelt sandbox), but excellent design reference.

- **CyberArk Secretless Broker** (github.com/cyberark/secretless-broker) — Go,
  Apache-2.0, 368 stars.  Production-grade credential injection sidecar with
  `generic_http` connector (26 pre-built API configs).  Config-driven
  header/query/OAuth1 injection with URL pattern matching.  Enterprise-proven
  but in maintenance mode.  Not viable as dependency (Go, CyberArk ecosystem).

# Credential Proxy: MITM Approach (Alternative)

## Problem Statement

Same as the socket proxy design: task containers currently have access to real
credentials via shared config directory mounts, enabling exfiltration by
compromised agents.

This document describes the **MITM proxy alternative** — intercepting HTTPS
traffic at the network boundary, terminating TLS with a custom CA, and injecting
credentials before forwarding to upstream APIs.  This is the approach used by
Docker Sandboxes, E2B, Sandcat, and IronCurtain.

This design serves as a **fallback** for cases where the socket proxy approach
(see `credential-proxy-socket.md`) is not feasible — e.g. tools that don't
support base URL overrides, or environments where transparent interception is
preferred over per-tool configuration.

## When to Use This Instead of the Socket Approach

- A vendor tool hardcodes its API endpoint with no override mechanism
- The tool validates TLS certificate chains against known CAs (cert pinning)
  in a way that's broken by socket redirect but works with system CA injection
- Agent-agnostic mode is needed (unknown/untrusted tool in container)
- The added complexity of CA management and trust store injection is acceptable

## Design Goals

1. **Agent-agnostic** — works with any tool that speaks HTTPS, regardless of
   base URL support.
2. **Domain-based routing** — credentials injected by matching the destination
   domain, not per-tool env vars.
3. **Shared credential store** — same `~/.local/share/terok/credentials/`
   layout as the socket approach.
4. **Same auth container pattern** — credentials captured from temp containers,
   never in shared mounts.

## Architecture: Two Variants

### Variant A: Custom mitmproxy (simpler, sufficient for most cases)

```
┌─ HOST ─────────────────────────────────────────────────────────────┐
│                                                                     │
│  mitmproxy + credential addon                                       │
│    Listens on TCP port (e.g. 18080)                                 │
│    Mode: explicit (HTTPS_PROXY) or transparent (nftables redirect)  │
│    CA cert: generated per-host, persistent                          │
│    Addon: domain → credential mapping, header injection             │
│                                                                     │
│  Credential Store  (same as socket approach)                        │
│  ~/.local/share/terok/credentials/{project}/                        │
│                                                                     │
│  SSH Agent  (same as socket approach)                               │
│  /run/terok/ssh-agent-{project_id}.sock                             │
│                                                                     │
└──────────────────────────────────────────────────────────────────────┘
                       │
                       │ HTTPS_PROXY=http://host:18080
                       │ (or nftables DNAT for transparent)
                       ▼
┌─ TASK CONTAINER ────────────────────────────────────────────────────┐
│                                                                     │
│  CA cert injected into trust stores:                                │
│    /usr/local/share/ca-certificates/terok-proxy.crt                 │
│    NODE_EXTRA_CA_CERTS=/usr/local/share/ca-certificates/terok-…     │
│    SSL_CERT_FILE=/etc/ssl/certs/ca-certificates.crt                 │
│    REQUESTS_CA_BUNDLE=/etc/ssl/certs/ca-certificates.crt            │
│                                                                     │
│  Phantom API keys (to satisfy tool startup checks):                 │
│    ANTHROPIC_API_KEY=<phantom>                                      │
│    OPENAI_API_KEY=<phantom>                                         │
│    etc.                                                             │
│                                                                     │
│  Tool makes HTTPS request to api.anthropic.com →                    │
│    HTTPS_PROXY routes through mitmproxy →                           │
│    mitmproxy terminates TLS (custom CA) →                           │
│    addon replaces phantom with real credential →                    │
│    mitmproxy forwards to real api.anthropic.com (genuine TLS) →     │
│    response streams back through proxy                              │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

#### Proxy mode choice

**Explicit proxy** (`HTTPS_PROXY` env var):
- Simpler setup, no iptables needed
- Tool must respect `HTTPS_PROXY` (most do, but not all)
- A compromised agent could unset the env var to bypass the proxy
- Sufficient if the threat model is "accidental exfiltration by prompt injection"
  (agent follows instructions but doesn't actively try to circumvent proxy)

**Transparent proxy** (nftables/iptables DNAT):
- Requires `CAP_NET_ADMIN` on the host side (not in container)
- Cannot be bypassed from inside the container
- All TCP traffic is intercepted regardless of env vars
- Requires mitmproxy `--mode transparent` and `SO_ORIGINAL_DST` support
- Better security, more complex setup
- Can be done via shield's existing nftables infrastructure

**Recommendation:** Start with explicit proxy (`HTTPS_PROXY`).  Add transparent
mode later if bypass resistance is needed.  The agent tools we support all
respect `HTTPS_PROXY`.

#### mitmproxy addon

```python
"""Credential injection addon for terok MITM proxy."""
import json
from mitmproxy import http, ctx

class TerokCredentialInjector:
    """Inject real credentials into proxied requests."""

    def __init__(self):
        self.rules: dict = {}  # domain → {header_name, header_value}

    def load(self, loader):
        loader.add_option(
            "terok_credential_config", str, "",
            "Path to credential injection config JSON",
        )

    def configure(self, updated):
        if "terok_credential_config" in updated:
            path = ctx.options.terok_credential_config
            if path:
                with open(path) as f:
                    self.rules = json.load(f)

    def request(self, flow: http.HTTPFlow) -> None:
        host = flow.request.pretty_host
        for domain, cred in self.rules.items():
            if host == domain or host.endswith("." + domain):
                # Strip agent's phantom auth
                flow.request.headers.pop("Authorization", None)
                flow.request.headers.pop("X-Api-Key", None)
                flow.request.headers.pop("x-api-key", None)
                # Inject real credential
                flow.request.headers[cred["header"]] = cred["value"]
                return

    def responseheaders(self, flow: http.HTTPFlow) -> None:
        """Stream SSE and chunked responses without buffering."""
        ct = flow.response.headers.get("content-type", "")
        if "text/event-stream" in ct or "application/x-ndjson" in ct:
            flow.response.stream = True
        if flow.response.headers.get("transfer-encoding") == "chunked":
            flow.response.stream = True

addons = [TerokCredentialInjector()]
```

- Java/JVM: requires `keytool -importcert` into Java's keystore.  Not currently

### Variant B: WireGuard tunnel + mitmproxy (inspired by Sandcat)

[Sandcat](https://github.com/VirtusLab/sandcat) (VirtusLab, Apache-2.0) is a
Docker-based AI agent sandbox that routes all container traffic through a
WireGuard tunnel to a mitmproxy instance.  It is Docker-only and not portable
to Podman without a full rewrite, but its **design patterns** are valuable
inspiration for terok's own implementation.

#### Sandcat's key ideas worth adopting

**1. WireGuard as cryptographic network boundary**

Sandcat uses `mitmproxy --mode wireguard` — mitmproxy acts as a WireGuard
server.  The agent container is a WireGuard client with a kill-switch (iptables
DROP on all non-WireGuard egress).  The tunnel is the only way out.

This is stronger than iptables/nftables alone because:
- A process with `CAP_NET_ADMIN` could flush iptables rules — WireGuard's
  cryptographic authentication cannot be bypassed from the app container
- Per-container WireGuard keys enable identity-based routing (the proxy knows
  which container sent each request)

**2. Secret substitution with per-secret host allowlists (leak detection)**

Sandcat's most elegant feature: each secret gets a per-host allowlist:

```json
{
  "secrets": {
    "ANTHROPIC_API_KEY": {
      "value": "sk-ant-real-key",
      "hosts": ["api.anthropic.com"]
    }
  }
}
```

The container receives `ANTHROPIC_API_KEY=SANDCAT_PLACEHOLDER_ANTHROPIC_API_KEY`.
The mitmproxy addon:
- Scans request URLs, headers, and bodies for placeholder strings
- If the destination host matches the secret's `hosts` allowlist: replaces
  placeholder with real value
- If the destination does NOT match: **blocks with 403** (leak detection!)

This detects exfiltration attempts — if an agent embeds the placeholder in a
request to `github.com`, the proxy catches it.

**3. Network rules as ordered allow/deny with glob patterns**

Sandcat uses three-layer settings (user, project, local) with first-match-wins
network rules.  This is similar to terok-shield's domain allowlist but with
HTTP method filtering.

#### terok's own implementation (not a Sandcat port)

Rather than porting Sandcat (Docker-only, Docker Compose, VS Code-centric),
terok should implement the WireGuard + mitmproxy pattern natively:

```
┌─ HOST ──────────────────────────────────────────────────────┐
│                                                              │
│  mitmproxy (host process, not a container)                   │
│    --mode wireguard                                          │
│    + terok credential addon (inspired by SandcatAddon)       │
│    + per-secret host allowlists (leak detection)             │
│    + SSE streaming support                                   │
│                                                              │
│  WireGuard server keys generated per-host                    │
│  CA cert generated per-host                                  │
│                                                              │
└──────────────────────┬───────────────────────────────────────┘
                       │ WireGuard tunnel (UDP)
                       │
┌─ TASK CONTAINER ─────┴──────────────────────────────────────┐
│                                                              │
│  WireGuard client (wg0 interface)                            │
│    Configured during container init (then drop CAP_NET_ADMIN)│
│    Kill-switch: iptables DROP all non-WG egress              │
│                                                              │
│  CA cert in trust stores (system, Node.js, Python)           │
│  Phantom API keys as env vars                                │
│  ALL HTTPS traffic goes through tunnel → mitmproxy           │
│                                                              │
└──────────────────────────────────────────────────────────────┘
```

Key differences from Sandcat:
- **Single container** (not three) — WireGuard client setup runs as an init
  step inside the task container, then `CAP_NET_ADMIN` is dropped.  Alternatively,
  terok-shield's existing OCI hooks could configure the WireGuard interface from
  outside the container's namespace.
- **Host-side mitmproxy** (not containerized) — simpler lifecycle, no Docker
  Compose orchestration.
- **Podman-native** — uses Podman's pasta/slirp4netns networking, terok-shield's
  nftables hooks.
- **terok's credential store** — shared JSON-based store, not Sandcat's settings
  layers.

Integration with terok-shield: the shield's OCI hooks could be extended to set
up WireGuard interfaces and kill-switch rules during container startup, similar
to how shield already configures nftables rules via `pre_start()` annotations.

# Credential Proxy: Socket Approach (Primary)

## Problem Statement

terok currently bind-mounts vendor tool configuration directories (containing API
keys, OAuth tokens, SSH private keys) into task containers.  A prompt-injected or
supply-chain-compromised agent can read these secrets and exfiltrate them to any
allowed domain (e.g. `github.com`, which shield cannot block without degrading
functionality).

Docker-based sandboxes solve this with MITM proxies that inject credentials at
the network boundary.  This document proposes an alternative that preserves TLS
integrity: a **Unix-socket-based credential proxy** that leverages documented
vendor env vars (`ANTHROPIC_BASE_URL`, `OPENAI_BASE_URL`, `ANTHROPIC_UNIX_SOCKET`,
gh `http_unix_socket`, etc.) to redirect API traffic through a host-side reverse
proxy that injects real credentials.

## Design Goals

1. **No real secret enters a task container** — not in env vars, not on
   the filesystem, not even transiently.
2. **Single subsystem** covering API-key auth, OAuth, and CLI tool auth (gh/glab).
3. **Full shared mounts preserved** — settings, memories, plans stay read-write
   shared.  Shared mounts are safe because auth never flows through them.
4. **Per-project credentials** — different projects can use different accounts.
5. **Vendor-supported mechanisms only** — documented env vars, no TLS interception.

## Auth Flow

### Two kinds of containers

**Auth containers** (ephemeral, one-shot):

1. `terokctl auth {project} {provider}` creates a temporary container
2. Config dir is mounted to a **temp directory** (not the shared mount)
3. Vendor auth command runs interactively (browser OAuth, API key prompt, etc.)
4. terok captures the credential file from the temp directory
5. terok stores the credentials in its own credential store
6. Temp directory is deleted
7. The shared config mount **never sees real credentials**

**Task containers** (long-lived):

1. Shared config dirs mounted as before (settings, memories, plans)
2. Phantom API key env vars injected
3. Base URL env vars point to proxy socket
4. Agent operates in "API key mode" — never touches OAuth, never reads credential files
5. All API calls routed through proxy

### OAuth token refresh

The proxy handles refresh internally.  When an access token expires:

```
Proxy (host-side, in-memory):
  1. POST to provider's token endpoint with real refresh_token
     (e.g. console.anthropic.com/v1/oauth/token for Claude)
  2. Receive new access_token + refresh_token
  3. Update credential store
  4. Use new access_token for subsequent forwarded requests
  Container is never involved in this flow.
```

Refresh endpoints per provider:

| Provider   | Token endpoint                                    | Notes                        |
|------------|---------------------------------------------------|------------------------------|
| Claude     | `console.anthropic.com/v1/oauth/token`            | Rotating refresh tokens      |
| Codex      | OpenAI OAuth endpoint (standard OIDC)             | `CODEX_REFRESH_TOKEN_URL_OVERRIDE` exists |
| gh         | `github.com/login/oauth/access_token`             | Or uses long-lived PAT       |
| Others     | API key (no refresh needed)                       |                              |

## SSH Agent Isolation

Per-project, fully isolated from the user's SSH agent:

```python
# Host-side, managed by terok/terok-sandbox
agent_sock = f"/run/terok/ssh-agent-{project_id}.sock"

# Start dedicated agent (never touches user's SSH_AUTH_SOCK)
subprocess.run(["ssh-agent", "-a", agent_sock])

# Load only this project's key, with TTL
env = {"SSH_AUTH_SOCK": agent_sock}
subprocess.run(["ssh-add", "-t", str(ttl_seconds), key_path], env=env)

# Mount socket into container
volumes.append(f"{agent_sock}:/run/terok/ssh-agent.sock")
container_env["SSH_AUTH_SOCK"] = "/run/terok/ssh-agent.sock"

# On task teardown: kill agent, remove socket
```

The private key never enters the container.  The SSH agent provides a signing
oracle only — the container can authenticate to SSH servers but cannot extract
the key.

## Phantom Token Registry

Extends the gate token pattern.  Per-task phantom tokens map to
(project, credential_set):

```json
{
  "a1b2c3d4e5f6...": {
    "project": "myproject",
    "task": "task-001",
    "credential_set": "default"
  }
}
```

Storage: `~/.local/share/terok/state/proxy/tokens.json`

Lifecycle:
- Created when task environment is built
- Passed to container as `ANTHROPIC_API_KEY`, `OPENAI_API_KEY`, etc.
- Validated by proxy on every request
- Revoked on task deletion

## Proxy Implementation

### Transport

Unix domain socket: `/run/terok/credential-proxy.sock`

The proxy speaks HTTP over the socket.  This is **not** a TCP server — no port
allocation, no network visibility.  The vendor tools send standard HTTP
requests; the socket is just the transport layer.

### Core logic (aiohttp)

```python
# Simplified — real implementation adds error handling, logging, config
async def handle_request(request: web.Request) -> web.StreamResponse:
    # 1. Extract and validate phantom token
    phantom = extract_phantom_token(request)
    task_info = token_registry.lookup(phantom)
    if not task_info:
        return web.Response(status=401, text="Invalid token")

    # 2. Route by path prefix
    provider, upstream_path = parse_route(request.path)
    if not provider:
        return web.Response(status=404, text="Unknown route")

    # 3. Load real credential
    cred = credential_store.get(task_info.credential_set, provider)
    real_token = await cred.get_valid_token()  # refreshes if expired

    # 4. Build upstream request
    upstream_url = provider.upstream_base + upstream_path
    headers = dict(request.headers)
    headers[provider.auth_header_name] = provider.auth_header_value(real_token)
    # Strip phantom token from forwarded headers
    headers.pop("X-Api-Key", None)
    headers.pop("Authorization", None)

    # 5. Forward and stream response
    async with aiohttp.ClientSession() as session:
        async with session.request(
            request.method, upstream_url,
            headers=headers, data=request.content,
        ) as upstream_resp:
            resp = web.StreamResponse(
                status=upstream_resp.status,
                headers=upstream_resp.headers,
            )
            await resp.prepare(request)
            async for chunk in upstream_resp.content.iter_any():
                await resp.write(chunk)
            await resp.write_eof()
            return resp
```

### Lifecycle

Mirrors the gate server pattern:

- **Primary**: systemd socket activation on the Unix socket path
- **Fallback**: managed daemon (PID file, SIGTERM stop)
- **Pre-task check**: `ensure_proxy_reachable()` before task creation
- **Shield integration**: proxy socket doesn't need a port in `loopback_ports`
  (Unix socket, not TCP), but if TCP fallback is used, add the port

### CLI commands

```
terokctl credential-proxy install   # install systemd units
terokctl credential-proxy start     # start daemon (fallback)
terokctl credential-proxy stop      # stop daemon
terokctl credential-proxy status    # check reachability
```

## Security Properties and Limitations

### What this design prevents

- **Secret exfiltration**: real API keys, OAuth tokens, and SSH private keys
  never enter the task container.  There is nothing to steal.
- **Phantom token abuse**: phantom tokens are per-task, bound to localhost,
  and revoked on task deletion.  Exfiltrating them is pointless.

### What this design does NOT prevent (by design)

**Proxy bypass by a hijacked agent.**  A compromised agent could unset the
base URL env vars (`ANTHROPIC_BASE_URL`, `OPENAI_BASE_URL`, etc.) and attempt
to use the tools with different authentication (e.g. its own hardcoded key,
or no auth at all).  This is **not a credential exfiltration risk** — if the
agent bypasses the proxy, it also loses access to the user's credentials.

The risk profile of a proxy-bypassing agent is equivalent to:
- Visiting a random website (which shield already gates)
- Accessing the user's repo without proper authentication (which is the
  endpoint operator's concern, not ours)

If bypass resistance is required, the MITM proxy alternative with transparent
mode or WireGuard tunneling (see `credential-proxy-mitm.md`) provides
stronger guarantees at the cost of TLS integrity and additional complexity.

### Leak detection (optional, inspired by Sandcat)

The proxy can validate that phantom tokens are only used on their intended
routes.  If a phantom token for `ANTHROPIC_API_KEY` appears in a request to
the `/gh/` route, the proxy can flag or block it as a potential exfiltration
attempt.  This only protects against leaks via tools that actually use the
proxy — a hijacked agent that bypasses the proxy entirely is outside this
scope (but also has no real credentials to leak).

## Known Risks and Open Questions

### Risk: Tool refuses to start without credential file

Some tools may check for the presence of a credential file at startup, not just
an API key env var.  Mitigation: test each tool with only the phantom API key
env var set and no credential file.  If a tool refuses, provide a stub credential
file with the phantom token.

### Risk: Tool hardcodes OAuth endpoints

If a tool's OAuth refresh bypasses `ANTHROPIC_BASE_URL` (calls
`console.anthropic.com` directly), the container would need network access to
Anthropic's OAuth endpoint.  Verified mitigations:
- Claude: `CLAUDE_CODE_CUSTOM_OAUTH_URL` redirects OAuth
- Codex: `CODEX_REFRESH_TOKEN_URL_OVERRIDE` redirects refresh
- But with phantom API key mode, OAuth never triggers — the tool uses API key
  auth and skips OAuth entirely.

### Risk: Streaming performance

The proxy must handle SSE (Server-Sent Events) without buffering.  aiohttp's
`iter_any()` streams chunks as they arrive — this is the same pattern used by
the gate server for git pack data.

### Risk: Vendor tool update breaks base URL support

If a vendor removes or changes their base URL env var, the proxy stops working
for that tool.  Mitigation: these are documented, stable integration points used
by thousands of corporate proxy/gateway deployments.  Breaking them would be a
major regression for the vendor.

### Risk: Unix socket permissions in rootless Podman

The socket file must be readable/writable by the container's UID after user
namespace mapping.  Mitigation: set socket permissions to match container UID,
or use `--userns=keep-id` (which terok already does).

### Open: Where does this live? (terok vs terok-sandbox)

The proxy is container-runtime plumbing (like the gate server).  If we are
extracting container-level concerns to terok-sandbox, the proxy should live
there.  The credential store and auth interceptor stay in terok (project-level
concerns).

### Open: Per-project vs per-task sockets

Current design: one daemon, one socket, per-task phantom tokens.
Alternative: per-task sockets for stronger isolation.
Recommendation: start with one daemon (simpler), add per-task sockets later if
needed.

### Open: Vibe config file vs env var

Vibe has no `MISTRAL_BASE_URL` env var — the `api_base` must be set in
`config.toml` under a provider entry.  terok already writes per-task config;
the proxy URL would be injected there.  Needs testing to confirm Vibe accepts
HTTP localhost as `api_base`.

## Implementation Plan

1. **Credential store** — JSON files, fcntl locks, per-project structure
2. **Auth interceptor** — modify `terokctl auth` to use temp mounts, capture
   credentials to the store

4. **Proxy core** — aiohttp Unix socket server, path routing, header injection
5. **OAuth refresh** — per-provider refresh logic in proxy
